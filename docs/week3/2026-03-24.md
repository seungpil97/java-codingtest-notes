## 문제 1 - 삽입 정렬 직접 구현

### 문제
정수 배열이 주어졌을 때, 삽입 정렬을 직접 구현해서 오름차순으로 정렬 후 출력하시오.  
(`Arrays.sort()` 사용 금지)

### 풀이 아이디어
현재 값(`key`)을 앞쪽 정렬된 구간에서 올바른 위치를 찾아 삽입했다.  
`key`보다 큰 값들을 오른쪽으로 한 칸씩 밀고, 빈 자리에 `key`를 넣었다.

### 사용한 개념
- 이중 반복문 (for + while)
- `key` 변수로 현재 삽입할 값 저장
- `Arrays.stream().mapToInt().toArray()` — Stream 파싱

### 시간복잡도
- O(n²)

### 공간복잡도
- O(1)

---

### 버그 기록

#### 선택 정렬을 삽입 정렬로 착각
```java
// 버그 코드 — 선택 정렬 방식
for (int i = 0; i < arr.length; i++) {
    for (int j = i + 1; j < arr.length; j++) {
        if (arr[i] > arr[j]) {
            int swapValue = arr[i];
            arr[i] = arr[j];
            arr[j] = swapValue;
        }
    }
}

// 수정 코드 — 삽입 정렬 방식
for (int i = 1; i < arr.length; i++) {
    int key = arr[i];
    int j = i - 1;
    while (j >= 0 && arr[j] > key) {
        arr[j + 1] = arr[j];
        j--;
    }
    arr[j + 1] = key;
}
```

**교훈:** 세 가지 정렬의 차이를 명확히 구분해야 한다.
- 버블 정렬 → 인접한 두 원소 비교해서 큰 수를 뒤로
- 선택 정렬 → 최솟값 인덱스 찾아서 맨 앞과 swap
- 삽입 정렬 → 현재 값을 앞쪽 정렬된 구간의 올바른 위치에 삽입

---

### 삽입 정렬 동작 원리

```
[3, 1, 4, 1, 5]

i=1, key=1 → arr[0]=3>1 → 3을 오른쪽으로 → [_, 3, 4, 1, 5] → 삽입 → [1, 3, 4, 1, 5]
i=2, key=4 → arr[1]=3<4 → 그대로              → [1, 3, 4, 1, 5]
i=3, key=1 → arr[2]=4>1, arr[1]=3>1, arr[0]=1>1? No → [1, 1, 3, 4, 5]
i=4, key=5 → arr[3]=4<5 → 그대로              → [1, 1, 3, 4, 5]
```

---

### 최종 코드
```java
br.readLine();
int[] arr = Arrays.stream(br.readLine().split(" "))
    .mapToInt(Integer::parseInt)
    .toArray();

for (int i = 1; i < arr.length; i++) {
    int key = arr[i];
    int j = i - 1;

    while (j >= 0 && arr[j] > key) {
        arr[j + 1] = arr[j];
        j--;
    }
    arr[j + 1] = key;
}
```

### 면접 질문 1. 삽입 정렬이 어떻게 동작하는지 설명해보세요.

#### 답변
현재 값을 `key`에 저장하고, 앞쪽 정렬된 구간에서 `key`보다 큰 값들을 오른쪽으로 한 칸씩 밀었습니다.  
밀기가 끝나면 빈 자리에 `key`를 삽입합니다.  
카드를 한 장씩 뽑아 올바른 위치에 끼워 넣는 방식과 동일합니다.

---

### 면접 질문 2. 삽입 정렬이 다른 O(n²) 정렬보다 유리한 경우가 있나요?

#### 답변
거의 정렬된 배열에서 삽입 정렬은 O(n)에 가까운 성능을 보입니다.  
이미 정렬된 경우 while 루프가 거의 돌지 않기 때문입니다.  
반면 버블 정렬과 선택 정렬은 항상 O(n²)이라 이런 이점이 없습니다.

---

### 회고
삽입 정렬을 선택 정렬로 구현하는 실수를 했다.  
세 가지 정렬의 차이를 코드 수준에서 명확히 구분할 수 있어야 한다.  
Stream 파싱 `Arrays.stream().mapToInt().toArray()`을 처음 적용했고, N이 작을 때 유용하다는 것을 익혔다.

## 문제 2 - 수 정렬하기

### 문제
N개의 수가 주어졌을 때, 오름차순으로 정렬해서 출력하시오.  
※ 백준 2751번

### 풀이 아이디어
N이 최대 100만이므로 O(n²) 정렬은 시간 초과가 난다.  
`Arrays.sort()`(O(n log n))를 사용하고, 입출력도 최적화했다.

### 사용한 개념
- `StringTokenizer` — `split()`보다 빠른 파싱
- `Arrays.sort()` — O(n log n)
- `StringBuilder` — 출력 최적화

### 시간복잡도
- O(n log n)

### 공간복잡도
- O(n)

---

### N 크기별 알고리즘 선택 기준

| N 범위 | 허용 복잡도 | 파싱 | 정렬 | 출력 |
|---|---|---|---|---|
| N ≤ 10,000 | O(n²) 가능 | Stream 가능 | 어떤 정렬이든 | println 가능 |
| N ≥ 100,000 | O(n log n) 이하 | **StringTokenizer** | **Arrays.sort()** | **StringBuilder** |

---

### StringTokenizer vs split()

```java
// split() — 느림
String[] tokens = br.readLine().split(" ");

// StringTokenizer — 빠름
StringTokenizer st = new StringTokenizer(br.readLine());
int value = Integer.parseInt(st.nextToken());
```

---

### 출력 최적화

```java
// ❌ println() 100만번 호출 → 느림
for (int num : nums) {
    System.out.println(num);
}

// ✅ StringBuilder로 한 번에 출력
StringBuilder sb = new StringBuilder();
for (int num : nums) {
    sb.append(num).append("\n");
}
System.out.print(sb);
```

---

### 최종 코드
```java
int n = Integer.parseInt(br.readLine());
StringTokenizer st = new StringTokenizer(br.readLine());
int[] nums = new int[n];

for (int i = 0; i < n; i++) {
    nums[i] = Integer.parseInt(st.nextToken());
}

Arrays.sort(nums);

StringBuilder sb = new StringBuilder();
for (int num : nums) {
    sb.append(num).append("\n");
}
System.out.print(sb);
```

---

### 면접 질문 1. 왜 삽입 정렬 대신 Arrays.sort()를 사용했나요?

#### 답변
N이 최대 100만이기 때문입니다.  
삽입 정렬은 O(n²)이라 100만² = 1조 번 연산이 필요해서 시간 초과가 납니다.  
Arrays.sort()는 O(n log n)으로 2초 안에 통과할 수 있습니다.

---

### 면접 질문 2. 왜 StringBuilder를 사용했나요?

#### 답변
System.out.println()을 100만번 호출하면 I/O 비용이 매우 커집니다.  
StringBuilder에 모아서 한 번에 출력하면 I/O 호출이 1번으로 줄어들어 훨씬 빠릅니다.

---

### 회고
N의 크기를 보고 알고리즘을 선택하는 습관이 중요하다.  
N ≥ 100,000이면 입력(StringTokenizer), 정렬(Arrays.sort), 출력(StringBuilder) 세 가지 모두 최적화해야 한다.  
Stream 파싱은 편리하지만 N이 클 때는 오버헤드가 생길 수 있어서 for문 + StringTokenizer가 더 안전하다.

## 문제 3 - 카드 정렬하기

### 문제
카드 묶음 여러 개를 하나로 합칠 때, 두 묶음을 합치는 비용은 두 묶음의 합이다.  
최소 비용으로 모든 묶음을 합칠 때의 총 비용을 출력하시오.

### 풀이 아이디어
항상 가장 작은 두 묶음을 합치면 최소 비용이 된다.  
매번 가장 작은 값을 꺼내야 하므로 `PriorityQueue`를 사용했다.  
두 값을 꺼내서 합산하고 비용에 누적한 뒤, 합산한 값을 다시 넣었다.

### 사용한 개념
- `PriorityQueue` — 항상 가장 작은 값이 top
- `poll()` — 가장 작은 값 꺼내기
- `add()` — 합친 묶음 다시 넣기

### 시간복잡도
- O(n log n) — poll/add 각 O(log n), n번 반복

### 공간복잡도
- O(n)

---

### PriorityQueue란?

일반 Queue는 먼저 들어온 것이 먼저 나오지만, PriorityQueue는 값이 작은 것이 먼저 나온다.

```
add(5) → [5]
add(1) → [1, 5]   ← 자동으로 작은 값이 앞으로
add(3) → [1, 3, 5]

poll() → 1  (가장 작은 것)
poll() → 3
poll() → 5
```

| 연산 | 복잡도 |
|---|---|
| add() | O(log n) |
| poll() | O(log n) |
| peek() | O(1) |

---

### 동작 예시 (`1 2 3`)

```
pq: [1, 2, 3]

poll(1) + poll(2) = 3, totalCost=3,  pq: [3, 3]
poll(3) + poll(3) = 6, totalCost=9,  pq: [6]

출력: 9 ✅
```

---

### 최종 코드
```java
PriorityQueue<Integer> pq = new PriorityQueue<>();

for (int num : nums) {
    pq.add(num);
}

int totalCost = 0;

while (pq.size() > 1) {
    int first = pq.poll();
    int second = pq.poll();
    int merged = first + second;
    totalCost += merged;
    pq.add(merged);
}

System.out.println(totalCost);
```

---

### 왜 Arrays.sort()가 필요 없나요?

PriorityQueue 자체가 삽입 시 자동으로 정렬해주기 때문에  
`Arrays.sort()`로 미리 정렬할 필요가 없다.

---

### 면접 질문 1. 왜 항상 가장 작은 두 묶음을 합쳐야 최소 비용인가요?

#### 답변
큰 묶음을 먼저 합치면 그 합산값이 이후 연산에도 계속 포함되기 때문입니다.  
작은 것부터 합치면 큰 값이 비용에 포함되는 횟수를 최소화할 수 있습니다.  
이는 허프만 코딩과 같은 그리디 알고리즘의 대표적인 원리입니다.

---

### 면접 질문 2. PriorityQueue를 사용한 이유는 무엇인가요?

#### 답변
매번 가장 작은 두 값을 꺼내야 하기 때문입니다.  
배열로 구현하면 매번 정렬이 필요해서 O(n²)이 되지만,  
PriorityQueue는 poll()이 O(log n)이라 전체 O(n log n)으로 처리할 수 있습니다.

---

### 회고
PriorityQueue를 처음 사용했다. 일반 Queue와 달리 값이 작은 것이 먼저 나온다는 점이 핵심이다.  
PriorityQueue에 넣으면 자동으로 정렬되므로 Arrays.sort()가 불필요하다는 점도 배웠다.  
그리디 알고리즘의 핵심 — "매 순간 최선의 선택을 하면 전체 최적이 된다" 를 처음 경험했다.
