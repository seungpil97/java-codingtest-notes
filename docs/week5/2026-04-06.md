## 🟢 문제 1. 두 수의 합 (Two Sum)

### 문제 설명

정수 배열 `numbers`와 정수 `target`이 주어진다.
배열에서 두 수를 골라 더했을 때 `target`이 되는 두 인덱스를 출력하라.
항상 정답이 존재하며, 같은 원소를 두 번 사용할 수 없다.
인덱스는 작은 것부터 출력한다.

- 제약 조건: 2 ≤ N ≤ 100,000

### 문제 요약

- 한 줄 요약: 배열을 순회하면서 두 수를 골라 더했을 때 target이 되는 두 인덱스 출력
- 입력 / 출력 구조: 정수 N개짜리 배열 + target 입력 → 두 인덱스 출력
- 헷갈리는 조건: 인덱스는 작은 것부터 출력

### 풀이 전 제출 내용

- 입력 크기 (N): 100,000
- 목표 시간복잡도: O(n)
- 사용할 알고리즘 후보: HashMap 활용
- 선택한 이유: 키로 값을 O(1)에 찾을 수 있으니 배열을 한 번만 순회하면서 풀 수 있다


### 풀이 아이디어

- `num`을 순회하면서 `partner = target - num` 계산
- partner가 map에 있으면 정답 출력
- 없으면 `map.put(num, i)` 저장하고 다음으로

### 사용한 개념

- HashMap: key=숫자값, value=인덱스
- `map.containsKey()` — O(1) 조회
- `map.get()` — O(1) 인덱스 반환

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(n) — 배열 1회 순회
- 공간복잡도: O(n) — map 저장

### 버그 기록

1. `st.nextToken()` 두 번 호출 → 변수에 담아 재사용으로 수정
2. `map.get(i)` 로 조회 → `map.containsKey(partner)` / `map.get(partner)` 로 수정
3. key/value 반대로 저장 → `map.put(num, i)` 로 수정

### 최종 코드

```java
package week5;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.HashMap;
import java.util.Map;
import java.util.StringTokenizer;

public class April07TwoSum {

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    String[] input = br.readLine().split(" ");
    int n = Integer.parseInt(input[0]);
    int target = Integer.parseInt(input[1]);
    StringTokenizer st = new StringTokenizer(br.readLine());

    Map<Integer, Integer> map = new HashMap<>();

    for (int i = 0; i < n; i++) {
      int num = Integer.parseInt(st.nextToken());
      int partner = target - num;
      if (map.containsKey(partner)) {
        int partnerIndex = map.get(partner);
        System.out.println(partnerIndex + " " + i);
        break;
      }
      map.put(num, i);
    }
  }
}
```

### 💡 HashMap

**장점:**

- `containsKey()`, `get()`, `put()` 모두 평균 O(1) — 배열처럼 인덱스 접근과 동일한 속도로 키-값 쌍을 다룰
  수 있어서, 순회 중 과거 정보를 빠르게 조회해야 할 때 특히 강력합니다.

**단점:**

- 순서를 보장하지 않습니다. 키를 순서대로 꺼내야 하는 상황이라면 `LinkedHashMap` 또는 `TreeMap`이 적합합니다.
- 해시 충돌이 많아지면 최악의 경우 O(n)까지 느려질 수 있습니다.

**실무 사용처:**

- 카카오 서버에서 유저 세션을 userId를 key로 관리할 때 HashMap 구조를 사용합니다. 수백만 유저 중 특정 유저 세션을 O(1)로
  조회하기 위해서입니다.
- Java의 `HttpSession` 내부도 HashMap 기반으로 속성을 저장합니다.

### 회고

- map의 key/value 방향을 처음에 반대로 설정하는 실수를 했다.
- 배열 없이 변수 하나로 처리할 수 있다는 것을 배웠다.
- `containsKey()` 는 boolean 반환, `get()` 은 값 반환 — 구분 필수.

## 🟡 문제 2. 연속 부분 배열의 합

### 문제 설명

양의 정수로 이루어진 배열에서, 합이 정확히 `target`이 되는 연속 부분 배열의 개수를 구하라.

- 제약 조건: 1 ≤ N ≤ 100,000 / 1 ≤ numbers[i] ≤ 1,000

### 문제 요약 

- 한 줄 요약: 정수 N개짜리 배열에서 합이 정확히 target이 되는 연속 부분 배열 개수 출력
- 입력 / 출력 구조: 배열 크기 N, target 입력 → 개수 출력
- 헷갈리는 조건: "연속 부분 배열" — 시작점이 다르면 다른 배열로 카운트

### 풀이 전 제출 내용 

- 입력 크기 (N): 100,000
- 목표 시간복잡도: O(n)
- 사용할 알고리즘 후보: 투 포인터 (left, right)
- 선택한 이유: left, right 모두 오른쪽으로만 이동 → 배열 1번 순회

### 풀이 아이디어

- `right`를 늘리며 `sum += arr[right]`
- `sum > target` 이면 `sum -= arr[left]`, `left++` 반복 (while)
- `sum == target` 이면 `count++`

### 사용한 개념

- 투 포인터: left, right로 구간 조절
- while로 left를 여러 번 이동 — 양의 정수이기 때문에 성립

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(n) — left, right 각각 최대 n번 이동
- 공간복잡도: O(n) — arr 배열

### 버그 기록

1. `sum += arr[left]` → `sum += arr[right]` 수정
2. `if (sum > target)` → `while (sum > target)` 수정 — 한 번만 줄이면 부족한 케이스 처리

### 최종 코드

```java
package week5;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.StringTokenizer;

public class April07SubarraySum {
  static int arr[];

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    String[] input = br.readLine().split(" ");
    int n = Integer.parseInt(input[0]);
    int target = Integer.parseInt(input[1]);
    StringTokenizer st = new StringTokenizer(br.readLine());
    arr = new int[n];

    for (int i = 0; i < n; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }

    int count = 0;
    int left = 0;
    int right = 0;
    int sum = 0;

    while (right < n) {
      sum += arr[right];
      right++;

      while (sum > target) {
        sum -= arr[left];
        left++;
      }

      if (sum == target) {
        count++;
      }
    }

    System.out.println(count);
  }
}
```

### 💡 투 포인터 (Two Pointer)

**장점:**

- left, right 모두 오른쪽으로만 이동하기 때문에 O(n)으로 연속 구간 문제를 풀 수 있습니다. 이중 루프 O(n²) 대비
  N=100,000에서 10억 번 → 10만 번으로 줄어듭니다.

**단점:**

- 양의 정수일 때만 성립합니다. 음수가 포함되면 sum이 커졌다 작아졌다 하기 때문에 이 패턴을 쓸 수 없습니다.

**실무 사용처:**

- 네트워크 패킷 분석에서 특정 크기의 연속 데이터 구간을 탐색할 때 슬라이딩 윈도우(투 포인터의 변형)를 사용합니다.
- 백준 2003번 "수들의 합 2" 가 이 패턴의 대표 문제입니다.

### 회고

- `if → while` 변경이 핵심이었다. 조건을 만족할 때까지 반복해야 한다는 것을 명심.
- 양의 정수 조건이 투 포인터 성립의 전제임을 이해했다.

## 🟡 문제 3. 최장 연속 문자 (슬라이딩 윈도우)

### 문제 설명

소문자 영문자로 이루어진 문자열에서, 같은 문자가 최대 k번까지만 등장하도록 제한할 때 조건을 만족하는 가장 긴 연속 부분 문자열의 길이를
구하라.

- 제약 조건: 1 ≤ 문자열 길이 ≤ 100,000 / 1 ≤ k ≤ 100,000

### 문제 요약 

- 한 줄 요약: 문자열에서 각 문자가 k번 이하로 등장하는 가장 긴 연속 부분 문자열 길이 출력
- 입력 / 출력 구조: 문자열, k 입력 → 최대 길이 출력
- 헷갈리는 조건: "가장 긴" 구간 — 모든 문자가 k번 이하인 구간 중 최대

### 풀이 전 제출 내용

- 입력 크기 (N): 100,000
- 목표 시간복잡도: O(n)
- 사용할 알고리즘 후보: 투 포인터 + HashMap
- 선택한 이유: k번 초과 시 left 이동, 이하면 right 이동 → 배열 1번 순회

### 풀이 아이디어

- `right` 문자를 map에 횟수 +1
- 해당 문자가 k 초과하면 `left` 문자 횟수 -1, `left++` 반복 (while)
- `maxLength = Math.max(maxLength, right - left)` 갱신

### 사용한 개념

- 슬라이딩 윈도우: 윈도우 안 문자 횟수를 HashMap으로 추적
- `map.getOrDefault(c, 0)` — NPE 방지
- `right - left` — 현재 윈도우 길이 계산

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(n) — left, right 각각 최대 n번 이동
- 공간복잡도: O(1) — 소문자 26개만 저장

### 버그 기록

1. `map.get(c)++` → `map.getOrDefault(c, 0) + 1` 수정 (NPE 방지 + 올바른 갱신)
2. `if (map.get(c) > k)` → `while (map.get(c) > k)` 수정
3. `while (left <= right)` → `while (right < token.length())` 수정 (범위 초과 방지)
4. 불필요한 `import java.awt.image.ImagingOpException` 제거

### 최종 코드

```java
package week5;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.HashMap;
import java.util.Map;

public class April07LongestSubstring {

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    String[] input = br.readLine().split(" ");
    String token = input[0];
    int k = Integer.parseInt(input[1]);

    Map<Character, Integer> map = new HashMap<>();

    int left = 0;
    int right = 0;
    int maxLength = 0;

    while (right < token.length()) {
      char c = token.charAt(right);
      right++;
      map.put(c, map.getOrDefault(c, 0) + 1);

      while (map.get(c) > k) {
        char leftChar = token.charAt(left);
        map.put(leftChar, map.get(leftChar) - 1);
        left++;
      }

      maxLength = Math.max(maxLength, right - left);
    }

    System.out.println(maxLength);
  }
}
```

### 💡 슬라이딩 윈도우

**장점:**

- 윈도우를 늘리고 줄이는 방식으로 불필요한 재계산을 없애 O(n)에 풀 수 있습니다. left/right가 각각 최대 n번만 이동하기
  때문입니다.

**단점:**

- 윈도우 크기나 조건이 고정되지 않은 복잡한 문제에서는 적용이 어렵습니다. 조건이 여러 개 얽히면 left 이동 기준 설계가 까다로워집니다.

**실무 사용처:**

- TCP 흐름 제어의 슬라이딩 윈도우 프로토콜이 이 개념 그대로입니다. 수신자가 처리할 수 있는 만큼만 윈도우를 열어 데이터를 받습니다.
- Kafka에서 특정 시간 범위 내 메시지를 집계할 때 슬라이딩 윈도우 방식을 사용합니다.

### 회고

- `map.get(c)++` 이 map을 업데이트하지 않는다는 것을 실수로 배웠다. 항상 `map.put(key, value)` 로 명시적으로
  갱신해야 한다.
- left 이동 시 right 문자가 아닌 left 문자의 횟수를 줄여야 한다는 것을 주의.
- 윈도우 길이는 `right - left` 로 계산.
