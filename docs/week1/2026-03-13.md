## 문제 1 - 가장 많이 등장한 문자

### 문제
문자열이 하나 주어졌을 때, 가장 많이 등장한 문자를 출력하시오.  
단, 대소문자는 구분하지 않으며, 가장 많이 등장한 문자가 여러 개라면 먼저 등장한 문자를 출력하시오.

### 풀이 아이디어
입력 문자열을 먼저 소문자로 변환해 대소문자를 통일했다.  
`LinkedHashMap`을 사용해 처음 등장한 문자 순서를 유지하면서 각 문자의 등장 횟수를 누적했다.  
누적과 동시에 현재 문자의 count가 기존 최댓값보다 크면 `maxKey`와 `maxValue`를 갱신하는 방식으로 루프를 한 번만 돌았다.

### 사용한 개념
- `BufferedReader`
- `toLowerCase()`
- `LinkedHashMap`
- `getOrDefault()`
- `charAt()`
- 문자열 순회

### 시간복잡도
- O(n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### 최초 코드의 버그 — `maxKey` 초기화 누락
`maxValue`는 첫 번째 문자의 개수로 초기화했지만, `maxKey`는 `0`으로 초기화했다.  
첫 번째 문자가 최빈값인 경우 반복문에서 갱신이 한 번도 일어나지 않아 `maxKey`가 `0`인 채로 출력됐다.

```java
// 버그 코드
char maxKey = 0;  // maxValue와 기준이 불일치
int maxValue = countMap.get(input.charAt(0));

// 수정 코드
char maxKey = input.charAt(0);  // maxValue와 동일한 기준으로 초기화
int maxValue = countMap.get(input.charAt(0));
```

**교훈:** `maxKey`와 `maxValue`는 항상 같은 기준으로 초기화해야 한다.

---

### 최적화 — 루프 1회로 줄이기

최초 코드는 루프가 두 번이었다.
1. 문자열 순회 → `countMap` 구축
2. `countMap` 순회 → 최댓값 탐색

`countMap` 갱신 직후 바로 최댓값을 비교하면 루프 한 번으로 해결할 수 있다.

```java
char maxKey = input.charAt(0);
int maxValue = 0;

for (int i = 0; i < input.length(); i++) {
    char c = input.charAt(i);
    countMap.put(c, countMap.getOrDefault(c, 0) + 1);

    if (countMap.get(c) > maxValue) {
        maxKey = c;
        maxValue = countMap.get(c);
    }
}

System.out.println(maxKey);
```

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
입력 문자열을 `toLowerCase()`로 소문자로 통일한 뒤, `LinkedHashMap`을 사용해 각 문자의 등장 횟수를 누적했습니다.  
문자를 map에 저장할 때마다 현재 문자의 count가 기존 최댓값보다 크면 `maxKey`와 `maxValue`를 갱신했습니다.  
루프가 끝나면 `maxKey`에 가장 많이 등장한 문자가 저장되어 있으므로 그대로 출력했습니다.

---

### 면접 질문 2. 왜 `LinkedHashMap`을 사용했나요?

#### 답변
이 문제는 최빈값이 여러 개일 때 처음 등장한 문자를 출력해야 하기 때문입니다.  
일반 `HashMap`은 입력 순서를 보장하지 않지만, `LinkedHashMap`은 key가 들어온 순서를 유지합니다.  
입력 순서대로 순회하면서 count를 비교할 때 `>` 연산(엄격 비교)을 사용하면 동점이 발생해도 먼저 등장한 문자가 자동으로 유지됩니다.

---

### 면접 질문 3. 왜 동점일 때 처음 등장한 문자가 자동으로 유지되나요?

#### 답변
비교 조건이 `>`(엄격 비교)이기 때문입니다.  
같은 count가 나왔을 때 `>`는 `false`를 반환하므로 `maxKey`가 갱신되지 않고 이전 값이 유지됩니다.  
`>=`를 사용하면 나중에 등장한 문자로 덮어씌워지므로 조건 선택이 중요합니다.

---

### 면접 질문 4. 이 코드의 시간복잡도는 어떻게 되나요?

#### 답변
문자열 길이를 n이라고 하면 루프를 한 번만 순회하므로 O(n)입니다.  
각 문자마다 `LinkedHashMap` 조회와 저장이 평균 O(1)이기 때문에 전체 시간복잡도는 O(n)입니다.

---

### 면접 질문 5. 이 코드의 공간복잡도는 어떻게 되나요?

#### 답변
`LinkedHashMap`에 문자 종류별로 항목을 저장하므로 추가 공간이 필요합니다.  
최악의 경우 모든 문자가 서로 다르면 n개의 항목이 들어가므로 공간복잡도는 O(n)입니다.  
다만 알파벳 소문자만 입력된다는 조건이 있다면 최대 26개로 고정되어 O(1)로 볼 수도 있습니다.

---

### 회고
초기화 기준을 일치시키지 않아서 생긴 버그였다.  
`maxKey`와 `maxValue`는 항상 같은 기준으로 초기화해야 한다는 점을 확실히 익혔다.  
또한 누적과 최댓값 갱신을 루프 한 번에 처리하면 코드가 더 간결해진다는 것도 배웠다.

## 문제 2 - 문자열 압축

### 문제
문자열이 하나 주어졌을 때, 연속으로 같은 문자가 몇 번 나오는지 압축해서 출력하시오.  
단, 문자가 1번만 나오면 숫자는 생략한다.

### 풀이 아이디어
`LinkedHashMap`으로 접근했다가 "연속" 조건을 처리하지 못한다는 것을 확인했다.  
`LinkedHashMap`은 전체 등장 횟수를 세기 때문에, 같은 문자가 떨어져 있어도 합산되어 틀린 결과가 나온다.  
수정 후 `while` 루프로 연속 구간을 직접 세는 방식으로 변경했다.  
현재 문자와 같은 문자가 연속되는 동안 `count`를 증가시키고, 구간이 끝나면 문자와 count를 기록했다.

### 사용한 개념
- `BufferedReader`
- `while` 루프 (연속 구간 탐색)
- `charAt()`
- `StringBuilder`

### 시간복잡도
- O(n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### 최초 코드의 버그 — `LinkedHashMap`으로 전체 빈도 집계
`LinkedHashMap`은 같은 문자의 전체 등장 횟수를 합산하기 때문에 연속 구간을 구분하지 못한다.

```
입력: aabba

LinkedHashMap 방식 → {a=3, b=2} → 출력: a3b2  ❌
정답                              → 출력: a2b2a ✅
```

또한 count가 1일 때도 숫자를 붙이는 문제가 있었다 (`d` → `d1` 출력).

#### 수정 코드 — while 루프로 연속 구간 탐색
```java
int i = 0;
while (i < input.length()) {
    char c = input.charAt(i);
    int count = 0;

    while (i < input.length() && input.charAt(i) == c) {
        count++;
        i++;
    }

    sb.append(c);
    if (count > 1) sb.append(count);  // 1번이면 숫자 생략
}
```

**교훈:** "전체 빈도" 문제는 `Map`, "연속 구간" 문제는 `while` 루프로 직접 구간을 세야 한다.

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
외부 `while` 루프가 문자열 전체를 앞에서부터 순회합니다.  
현재 위치의 문자를 기준으로 내부 `while` 루프가 같은 문자가 연속되는 동안 `count`를 증가시키고 인덱스를 이동합니다.  
연속 구간이 끝나면 문자를 `StringBuilder`에 추가하고, `count`가 1보다 크면 숫자도 붙입니다.  
이 과정을 문자열 끝까지 반복하면 압축된 결과를 얻을 수 있습니다.

---

### 면접 질문 2. 왜 `LinkedHashMap`으로는 이 문제를 풀 수 없나요?

#### 답변
`LinkedHashMap`은 각 문자의 전체 등장 횟수를 저장하기 때문에 연속 여부를 구분하지 못합니다.  
예를 들어 `aabba`에서 `a`는 총 3번 등장하지만, `a2`와 `a1`로 나뉘어 압축되어야 합니다.  
이처럼 연속 구간이 핵심인 문제는 Map이 아닌 직접 구간을 추적하는 방식이 필요합니다.

---

### 면접 질문 3. 이 코드의 시간복잡도는 어떻게 되나요?

#### 답변
외부와 내부 while 루프가 중첩되어 있지만, 인덱스 `i`는 전체를 통틀어 한 번만 증가합니다.  
각 문자는 정확히 한 번씩만 처리되므로 전체 시간복잡도는 O(n)입니다.

---

### 회고
문제에 "연속"이라는 단어가 있으면 Map보다 while 루프로 구간을 직접 추적하는 패턴을 먼저 떠올려야 한다는 것을 배웠다.  
이 패턴은 RLE(런 길이 인코딩) 관련 문제에서 반복적으로 나오므로 확실히 익혀두어야 한다.

## 문제 3 - 중간값 찾기

### 문제
정수들이 공백으로 구분되어 주어졌을 때, 오름차순 정렬 후 중간값을 출력하시오.  
단, 정수의 개수는 홀수라고 가정한다.

### 풀이 아이디어
입력을 공백으로 나눈 문자열 배열을 `int[]`로 변환한 뒤 `Arrays.sort()`로 정렬했다.  
정렬 후 `nums.length / 2` 인덱스의 값이 중간값이다.

### 사용한 개념
- `BufferedReader`
- `split(" ")`
- `Integer.parseInt()`
- `Arrays.sort()`
- 중간 인덱스 계산

### 시간복잡도
- O(n log n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### 최초 코드의 버그 — `String[]` 그대로 정렬 (사전순 정렬)
`Arrays.sort()`를 `String[]`에 적용하면 사전순(문자열) 정렬이 적용된다.  
숫자가 두 자리 이상이 되면 반드시 틀린 결과가 나온다.

```
입력: 10 9 2 1 5

문자열 정렬 → ["1", "10", "2", "5", "9"]  ❌
숫자 정렬   → [1, 2, 5, 9, 10]            ✅
```

이번 테스트 케이스 `7 1 9 3 5`는 모두 한 자리 수라 우연히 맞았을 뿐이다.

#### 수정 코드
```java
String[] tokens = br.readLine().split(" ");
int[] nums = new int[tokens.length];

for (int i = 0; i < tokens.length; i++) {
    nums[i] = Integer.parseInt(tokens[i]);
}

Arrays.sort(nums);
System.out.println(nums[nums.length / 2]);
```

**교훈:** 숫자를 다룰 땐 반드시 `int[]`로 변환 후 정렬해야 한다. `String[]` 정렬은 두 자리 수 이상에서 반드시 틀린다.

---

### 최적화 — Stream 활용

`for`문 대신 Stream으로 변환하는 방식도 있다.  
동작은 동일하며 취향 차이이므로 어느 쪽이든 무방하다.

```java
int[] nums = Arrays.stream(tokens)
                   .mapToInt(Integer::parseInt)
                   .toArray();
```

---

### 더 빠른 방법 (심화) — QuickSelect

정렬 없이 평균 O(n)에 중간값을 찾는 QuickSelect 알고리즘이 존재한다.  
구현 난이도가 높아 현 단계에서는 불필요하며, 코딩테스트 입문 레벨에서는 정렬 방식으로 충분하다.

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
입력 한 줄을 공백 기준으로 나눠 문자열 배열로 받은 뒤, 각 값을 `Integer.parseInt()`로 정수 배열에 저장했습니다.  
`Arrays.sort()`로 오름차순 정렬한 뒤, `nums.length / 2` 인덱스의 값을 출력했습니다.  
정수의 개수가 홀수라는 조건이 있으므로 가운데 인덱스가 항상 정확히 하나 존재합니다.

---

### 면접 질문 2. 왜 `String[]`을 그대로 정렬하면 안 되나요?

#### 답변
`Arrays.sort()`를 `String[]`에 적용하면 사전순 정렬이 적용됩니다.  
사전순 정렬은 첫 번째 문자부터 비교하기 때문에 `"10"`이 `"2"`보다 앞에 오게 됩니다.  
두 자리 이상의 숫자가 포함된 경우 반드시 틀린 결과가 나오므로 `int[]`로 변환 후 정렬해야 합니다.

---

### 면접 질문 3. 이 코드의 시간복잡도는 어떻게 되나요?

#### 답변
`Arrays.sort()`는 Dual-Pivot Quicksort를 사용하므로 평균 O(n log n)입니다.  
입력 파싱은 O(n)이지만 정렬이 지배적이므로 전체 시간복잡도는 O(n log n)입니다.

---

### 면접 질문 4. 정렬보다 더 빠른 방법이 있나요?

#### 답변
네, QuickSelect 알고리즘을 사용하면 평균 O(n)에 중간값을 찾을 수 있습니다.  
다만 구현이 복잡하고 최악의 경우 O(n²)이 될 수 있어서, 일반적인 코딩테스트에서는 정렬 방식이 더 안정적이고 실용적입니다.

---

### 회고
테스트 케이스가 한 자리 수로만 구성되어 있어서 `String[]` 정렬이 우연히 통과했지만, 두 자리 이상의 숫자가 있으면 바로 틀렸을 것이다.  
숫자 데이터는 반드시 `int[]`로 변환 후 정렬한다는 원칙을 꼭 기억해야 한다.  
또한 `import java.lang.reflect.Array`처럼 사용하지 않는 import가 남아있지 않도록 코드를 정리하는 습관도 중요하다.
