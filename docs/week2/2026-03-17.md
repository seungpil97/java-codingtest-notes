## 문제 1 - 괄호 유효성 검사

### 문제
문자열이 하나 주어졌을 때, 괄호가 올바르게 열리고 닫혔는지 판별하시오.  
올바르면 `YES`, 아니면 `NO`를 출력하시오.

### 풀이 아이디어
`(` 를 만나면 count를 증가시키고, `)` 를 만나면 count를 감소시키는 방식으로 구현했다.  
단, `)` 를 만났을 때 count가 0이면 닫을 괄호가 없다는 뜻이므로 즉시 `NO`를 출력하고 종료했다.  
루프가 끝난 후 count가 0이면 모든 괄호가 올바르게 짝지어진 것이므로 `YES`를 출력했다.

### 사용한 개념
- `BufferedReader`
- `charAt()`
- count 변수로 괄호 쌍 추적
- 조기 종료 (`return`)

### 시간복잡도
- O(n)

### 공간복잡도
- O(1)

---

### 버그 기록

#### 버그 1 — 출력값이 `true` / `false`
```java
// 버그 코드
System.out.println(false);
System.out.println(count == 0);

// 수정 코드
System.out.println("NO");
System.out.println(count == 0 ? "YES" : "NO");
```

#### 버그 2 — `sb.isEmpty()` 체크 오류
`sb.isEmpty()`는 아무 문자도 들어오지 않았을 때만 `true`다.  
한 번이라도 문자가 추가되면 이후엔 절대 `true`가 되지 않는다.

```
입력: ())(

( → count=1
) → sb 비어있지 않음 → count=0
) → sb 비어있지 않음 → count=-1  ← 여기서 잡았어야 함
( → count=0

count == 0 → YES 출력  ❌  (정답은 NO)
```

체크해야 할 건 `sb`가 비어있는지가 아니라 현재 열린 괄호가 없는지, 즉 `count == 0`이다.

```java
// 버그 코드
if (c == ')' && sb.isEmpty())

// 수정 코드
if (c == ')' && count == 0)
```

**교훈:** `)` 를 만났을 때 체크할 것은 "현재 매칭 대기 중인 `(` 가 있는가" = `count == 0` 이면 닫을 괄호가 없다는 뜻이다.

---

### 최종 코드
```java
int count = 0;
int i = 0;
while (i < input.length()) {
    char c = input.charAt(i);

    if (c == ')' && count == 0) {
        System.out.println("NO");
        return;
    }

    if (c == '(') count++;
    else count--;

    i++;
}
System.out.println(count == 0 ? "YES" : "NO");
```

---

### Stack을 쓰지 않은 이유 — count로 충분한 경우

이 문제는 `( )` 한 종류라서 count 변수 하나로 해결할 수 있다.  
하지만 `( ) [ ] { }` 세 종류가 섞이면 count 방식은 무너진다.

```
입력: ([)]

count 방식 → ( +1, [ +1, ) -1, ] -1 → count=0 → YES  ❌
Stack 방식 → ( push, [ push, ) 꺼내보니 [ → 불일치 → NO  ✅
```

count는 개수만 세고 순서를 기억하지 못한다.  
여러 종류의 괄호는 어떤 괄호가 열려있는지 기억해야 하므로 LIFO 구조가 필요하다.

| 상황 | 풀이 |
|---|---|
| `( )` 한 종류 | count 변수로 충분 |
| `( ) [ ] { }` 여러 종류 | LIFO 구조 필수 |

**교훈:** 자료구조 이름보다 "왜 LIFO가 필요한가"를 설명할 수 있는 게 더 중요하다.

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
문자열을 앞에서부터 순회하면서 `(`를 만나면 count를 증가시키고, `)`를 만나면 count를 감소시켰습니다.  
단, `)`를 만났을 때 count가 0이면 닫을 괄호가 없다는 뜻이므로 즉시 `NO`를 출력하고 종료했습니다.  
루프가 끝난 후 count가 0이면 모든 괄호가 올바르게 짝지어진 것이므로 `YES`를 출력했습니다.

---

### 면접 질문 2. 왜 Stack을 사용하지 않았나요?

#### 답변
이 문제는 `( )` 한 종류의 괄호만 사용하기 때문에 열린 괄호의 개수만 추적하면 충분합니다.  
count 변수 하나로 O(1) 공간으로 해결할 수 있어서 Stack을 사용하지 않았습니다.  
다만 `( ) [ ] { }` 여러 종류가 섞이면 어떤 괄호가 열려있는지 기억해야 하므로 그 경우엔 Stack이 필요합니다.

---

### 면접 질문 3. 이 코드의 시간복잡도와 공간복잡도는 어떻게 되나요?

#### 답변
문자열을 한 번만 순회하므로 시간복잡도는 O(n)입니다.  
추가로 사용하는 변수는 count, i뿐이므로 공간복잡도는 O(1)입니다.

---

### 회고
`sb.isEmpty()`와 `count == 0`의 차이를 정확히 이해하지 못해서 버그가 생겼다.  
또한 count 방식과 Stack 방식의 차이 — count는 개수만 세고, Stack은 순서까지 기억한다는 점이 핵심이다.  
괄호 종류가 하나냐 여러 개냐에 따라 풀이 방식을 선택하는 기준을 익혔다.

## 문제 2 - 스택으로 문자열 뒤집기

### 문제
문자열이 하나 주어졌을 때, Stack을 직접 사용해서 문자열을 뒤집어 출력하시오.  
(`StringBuilder.reverse()` 사용 금지)

### 풀이 아이디어
문자열을 앞에서부터 순회하면서 각 문자를 Stack에 `push()`했다.  
Stack은 LIFO 구조이기 때문에 `pop()`으로 꺼내면 자동으로 역순이 된다.  
꺼낸 문자를 `StringBuilder`에 추가해서 최종 결과를 출력했다.

### 사용한 개념
- `BufferedReader`
- `Deque<Character>` / `ArrayDeque`
- `push()` / `pop()`
- LIFO (Last In First Out)
- `StringBuilder`

### 시간복잡도
- O(n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### `pop()` 대신 `for-each`로 순회
`ArrayDeque`는 `push()`할 때 앞(head)에 쌓이고, `for-each`는 head부터 순회해서 우연히 역순이 된다.  
하지만 이건 `ArrayDeque` 내부 구조에 의존한 것이라 Stack의 LIFO 원리를 직접 활용한 게 아니다.

```java
// 버그 코드 — for-each 순회에 의존
for (Character c : stack) {
    sb.append(c);
}

// 수정 코드 — LIFO 원리 직접 활용
while (!stack.isEmpty()) {
    sb.append(stack.pop());
}
```

**교훈:** Stack 문제는 반드시 `push()` / `pop()`으로 풀어야 LIFO 원리를 제대로 쓴 것이다. 우연히 맞는 코드와 원리를 이해한 코드는 다르다.

---

### 최종 코드
```java
Deque<Character> stack = new ArrayDeque<>();

for (int i = 0; i < input.length(); i++) {
    stack.push(input.charAt(i));
}

StringBuilder sb = new StringBuilder();
while (!stack.isEmpty()) {
    sb.append(stack.pop());
}

System.out.println(sb);
```

### 흐름
```
push: h → e → l → l → o  (스택 top: o)
pop:  o → l → l → e → h  → "olleh" ✅
```

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
문자열을 앞에서부터 순회하면서 각 문자를 `push()`로 Stack에 쌓았습니다.  
Stack은 LIFO 구조이기 때문에 `pop()`으로 꺼내면 마지막에 넣은 문자부터 나옵니다.  
꺼낸 문자를 `StringBuilder`에 추가하면 자연스럽게 뒤집힌 문자열이 만들어집니다.

---

### 면접 질문 2. 왜 LIFO 구조가 문자열 뒤집기에 적합한가요?

#### 답변
문자열을 뒤집으려면 마지막 문자부터 읽어야 합니다.  
Stack에 앞에서부터 순서대로 push하면 마지막 문자가 top에 위치하게 됩니다.  
pop()으로 꺼내면 top부터, 즉 마지막 문자부터 꺼내지므로 자동으로 역순이 됩니다.

---

### 면접 질문 3. `java.util.Stack` 대신 `ArrayDeque`를 쓴 이유가 있나요?

#### 답변
`java.util.Stack`은 `Vector`를 상속받아 불필요한 동기화 오버헤드가 있습니다.  
반면 `ArrayDeque`는 더 가볍고 성능이 좋아서 단일 스레드 환경에서는 `ArrayDeque`를 Stack처럼 사용하는 것이 권장됩니다.  
`push()`, `pop()`, `isEmpty()` 모두 동일하게 사용할 수 있습니다.

---

### 회고
코드 결과는 맞았지만 LIFO 원리를 직접 활용하지 않았다는 점에서 수정이 필요했다.  
`pop()`을 사용해야만 "마지막에 넣은 것부터 꺼낸다"는 Stack의 핵심 원리를 제대로 쓴 것이다.  
우연히 맞는 코드와 원리를 이해한 코드는 다르다는 점을 기억해야 한다.

## 문제 3 - 큐로 프린터 순서 시뮬레이션

### 문제
N개의 문서가 순서대로 큐에 들어있고, 각 문서는 우선순위(1~9, 높을수록 우선)를 가진다.  
프린터 규칙에 따라 M번째로 넣은 문서(0-index)가 몇 번째로 출력되는지 구하시오.

**프린터 규칙**
1. 큐에서 문서를 꺼낸다
2. 큐에 남은 문서 중 더 높은 우선순위가 있으면 → 맨 뒤에 다시 넣는다
3. 없으면 → 해당 문서를 출력한다

### 풀이 아이디어
같은 우선순위 값이 여러 개일 수 있어서 우선순위 값만으로는 M번 문서를 추적할 수 없다.  
큐에 `{우선순위, 원래인덱스}` 쌍을 저장해서 M번 문서를 추적했다.  
꺼낸 문서보다 높은 우선순위가 큐에 있으면 맨 뒤로 다시 넣고, 없으면 count를 증가시킨 후 index가 M과 같은지 확인했다.

### 사용한 개념
- `BufferedReader`
- `Queue<int[]>` / `LinkedList`
- `poll()` / `add()`
- `{우선순위, 원래인덱스}` 쌍으로 문서 추적
- 큐 시뮬레이션

### 시간복잡도
- O(n²) — 매 단계마다 큐 전체를 순회해서 최댓값 확인

### 공간복잡도
- O(n)

---

### 핵심 아이디어 — `{우선순위, 원래인덱스}` 쌍으로 저장

우선순위 값만 넣으면 같은 값이 여러 개일 때 어떤 문서인지 구분할 수 없다.  
원래 인덱스를 함께 저장해서 M번 문서인지 확인할 수 있다.

```java
queue.add(new int[]{Integer.parseInt(secondLine[i]), i});  // {우선순위, 원래인덱스}

// 꺼낼 때 M번 문서인지 확인
if (current[1] == m) {
    System.out.println(count);
    return;
}
```

---

### 시뮬레이션 (`4 2` / `1 2 3 4`, M=2 추적)

```
시작: [{1,0}, {2,1}, {3,2}, {4,3}]

1. {1,0} 꺼냄 → 2,3,4 더 높음 → 맨 뒤로
2. {2,1} 꺼냄 → 3,4 더 높음   → 맨 뒤로
3. {3,2} 꺼냄 → 4 더 높음     → 맨 뒤로
4. {4,3} 꺼냄 → 없음 → count=1  (index3, M 아님)
5. {1,0} 꺼냄 → 2,3 더 높음   → 맨 뒤로
6. {2,1} 꺼냄 → 3 더 높음     → 맨 뒤로
7. {3,2} 꺼냄 → 없음 → count=2  (index2 == M=2) → 출력: 2 ✅
```

---

### 최종 코드
```java
String[] firstLine = br.readLine().split(" ");
int n = Integer.parseInt(firstLine[0]);
int m = Integer.parseInt(firstLine[1]);

String[] secondLine = br.readLine().split(" ");

Queue<int[]> queue = new LinkedList<>();
for (int i = 0; i < n; i++) {
    queue.add(new int[]{Integer.parseInt(secondLine[i]), i});
}

int count = 0;
while (!queue.isEmpty()) {
    int[] current = queue.poll();

    boolean hasHigher = false;
    for (int[] doc : queue) {
        if (doc[0] > current[0]) {
            hasHigher = true;
            break;
        }
    }

    if (hasHigher) {
        queue.add(current);
    } else {
        count++;
        if (current[1] == m) {
            System.out.println(count);
            return;
        }
    }
}
```

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
각 문서를 `{우선순위, 원래인덱스}` 쌍으로 큐에 저장했습니다.  
큐에서 문서를 꺼낸 뒤, 남은 문서 중 더 높은 우선순위가 있으면 맨 뒤에 다시 넣었습니다.  
없으면 count를 증가시키고, 꺼낸 문서의 원래 인덱스가 M과 같으면 count를 출력하고 종료했습니다.

---

### 면접 질문 2. 왜 우선순위 값만 저장하지 않고 인덱스를 같이 저장했나요?

#### 답변
같은 우선순위 값이 여러 개일 때 값만으로는 어떤 문서인지 구분할 수 없기 때문입니다.  
예를 들어 우선순위가 `3 1 3 2`일 때, 두 개의 `3` 중 어떤 게 M번 문서인지 알 수 없습니다.  
원래 인덱스를 함께 저장하면 `current[1] == m` 으로 정확히 추적할 수 있습니다.

---

### 면접 질문 3. 이 코드의 시간복잡도는 어떻게 되나요?

#### 답변
while 루프 안에서 매번 큐 전체를 순회해서 최댓값을 확인하기 때문에 O(n²)입니다.  
n이 최대 100으로 제한되어 있어서 이 문제에서는 충분히 효율적입니다.  
n이 커진다면 우선순위 큐(PriorityQueue)로 최댓값 확인을 O(log n)으로 줄일 수 있습니다.

---

### 면접 질문 4. `poll()`과 `add()`의 역할은 무엇인가요?

#### 답변
`poll()`은 큐의 맨 앞에서 문서를 꺼내고 제거하는 역할을 합니다.  
`add()`는 큐의 맨 뒤에 문서를 추가하는 역할을 합니다.  
이 두 연산으로 FIFO(First In First Out) 구조를 유지하면서 시뮬레이션을 구현할 수 있습니다.

---

### 회고
처음엔 입력 파싱 순서를 잘못 이해해서 N과 M을 뒤바꿔 읽었다.  
핵심은 우선순위 값만 저장하면 동일한 값이 여러 개일 때 추적이 불가능하다는 점이다.  
`{우선순위, 원래인덱스}` 쌍으로 저장하는 패턴은 큐 시뮬레이션 문제에서 자주 쓰이므로 기억해두어야 한다.
