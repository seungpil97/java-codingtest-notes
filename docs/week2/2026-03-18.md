## 문제 1 - 후위 표기식 계산

### 문제
공백으로 구분된 후위 표기식(Postfix)이 주어졌을 때, 계산 결과를 출력하시오.

```
일반식: 3 + 4       → 후위식: 3 4 +
일반식: (3 + 4) * 2 → 후위식: 3 4 + 2 *
```

### 풀이 아이디어
토큰을 앞에서부터 하나씩 읽으면서 숫자면 push, 연산자면 pop 두 번 해서 계산 후 다시 push했다.  
숫자인지 연산자인지는 `Character.isDigit(token.charAt(0))`으로 판별했다.  
루프가 끝나면 스택에 결과값 하나만 남으므로 `pop()`으로 출력했다.

### 사용한 개념
- `BufferedReader`
- `Deque<Integer>` / `ArrayDeque`
- `push()` / `pop()`
- `Character.isDigit()`
- `switch` 표현식 (Java 14+)

### 시간복잡도
- O(n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### 버그 1 — 숫자 판별 로직 오류
```java
// 버그 코드 — "+" 같은 연산자가 들어오면 Integer.parseInt("+") 에서 예외 발생
Character.isDigit(Integer.parseInt(token))

// 수정 코드
Character.isDigit(token.charAt(0))
```

#### 버그 2 — leftOperand / rightOperand 순서 오류
`+`, `*`는 순서가 달라도 결과가 같지만 `-`, `/`는 순서가 바뀌면 틀린다.
```java
// 버그 코드
int b = stack.pop();
int a = stack.pop();
calculator(token, b, a)  // 순서 뒤바뀜

// 수정 코드
int rightOperand = stack.pop();
int leftOperand = stack.pop();
calculate(token, leftOperand, rightOperand)
```

#### 버그 3 — 최종 출력 누락
```java
// 루프 끝나고 반드시 출력해야 함
System.out.println(stack.pop());
```

**교훈:** 후위 표기식에서 pop() 순서는 나중에 꺼낸 게 오른쪽 피연산자, 먼저 꺼낸 게 왼쪽 피연산자다.

---

### 최종 코드
```java
Deque<Integer> stack = new ArrayDeque<>();

for (String token : tokens) {
    if (Character.isDigit(token.charAt(0))) {
        stack.push(Integer.parseInt(token));
    } else {
        int rightOperand = stack.pop();
        int leftOperand = stack.pop();
        int result = calculate(token, leftOperand, rightOperand);
        stack.push(result);
    }
}

System.out.println(stack.pop());

private static int calculate(String operator, int leftOperand, int rightOperand) {
    return switch (operator) {
        case "+" -> leftOperand + rightOperand;
        case "-" -> leftOperand - rightOperand;
        case "*" -> leftOperand * rightOperand;
        case "/" -> leftOperand / rightOperand;
        default -> throw new IllegalArgumentException("Unknown operator: " + operator);
    };
}
```

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
토큰을 앞에서부터 하나씩 읽으면서 숫자면 스택에 push하고, 연산자면 스택에서 두 번 pop해서 계산한 뒤 결과를 다시 push했습니다.  
나중에 꺼낸 값이 오른쪽 피연산자, 먼저 꺼낸 값이 왼쪽 피연산자입니다.  
루프가 끝나면 스택에 최종 결과값 하나만 남으므로 그것을 출력했습니다.

---

### 면접 질문 2. 왜 후위 표기식 계산에 Stack이 적합한가요?

#### 답변
후위 표기식은 연산자를 만났을 때 직전에 들어온 두 숫자를 꺼내서 계산해야 합니다.  
가장 최근에 들어온 값을 먼저 꺼내는 LIFO 구조가 이 동작과 정확히 일치하기 때문에 Stack이 적합합니다.

---

### 면접 질문 3. `switch` 표현식에서 `default`에 예외를 던진 이유는 무엇인가요?

#### 답변
문제 조건상 유효한 연산자만 들어온다고 가정하지만, 그 외의 입력이 들어왔을 때 `return 0`처럼 잘못된 값을 반환하는 것보다 예외를 던지는 것이 더 명확하고 안전합니다.  
도달 불가능한 케이스는 예외 처리로 명시하는 것이 좋은 습관입니다.

---

### 회고
`Integer.parseInt(token)`으로 숫자 판별을 시도하다가 연산자에서 예외가 터지는 버그를 경험했다.  
숫자 판별은 `Character.isDigit(token.charAt(0))`으로 해야 한다는 점을 확실히 익혔다.  
또한 `a`, `b` 같은 의미 없는 변수명 대신 `leftOperand`, `rightOperand`처럼 역할이 드러나는 이름을 써야 한다.

## 문제 2 - 최솟값을 O(1)에 반환하는 스택

### 문제
push, pop, getMin 기능을 구현하되, `getMin()`은 O(1)이어야 한다.

### 풀이 아이디어
보조 스택(`minStack`)을 하나 더 사용해서 현재 최솟값을 항상 top에 유지했다.  
push할 때 새 값이 현재 최솟값보다 작거나 같으면 `minStack`에도 push했다.  
pop할 때 제거된 값이 현재 최솟값이면 `minStack`에서도 pop했다.  
`getMin()`은 `minStack.peek()`으로 O(1)에 최솟값을 반환했다.

### 사용한 개념
- `BufferedReader`
- 여러 줄 입력 처리 (`while readLine`)
- 보조 스택 패턴
- `peek()` — 꺼내지 않고 top만 확인

### 시간복잡도
- push / pop / getMin 모두 O(1)

### 공간복잡도
- O(n) — 보조 스택 추가 사용

---

### peek() vs pop() 차이

```
스택: [5, 3]  ← top이 3

pop()  → 3 반환 + 스택에서 제거  → 스택: [5]
peek() → 3 반환 + 스택 그대로    → 스택: [5, 3]
```

`getMin()`은 조회만 해야 하므로 `peek()` 사용이 필수다.

---

### 보조 스택 동작 예시

```
push 5 → mainStack: [5],    minStack: [5]
push 3 → mainStack: [5,3],  minStack: [5,3]  (3 <= 5이므로 minStack에도 push)
getMin → minStack.peek() = 3  ✅
pop    → mainStack: [5],    minStack: [5]     (3 제거됐으니 minStack에서도 제거)
getMin → minStack.peek() = 5  ✅
```

---

### 최종 코드
```java
private static final Deque<Integer> mainStack = new ArrayDeque<>();
private static final Deque<Integer> minStack = new ArrayDeque<>();

private static void push(int value) {
    mainStack.push(value);
    if (minStack.isEmpty() || value <= minStack.peek()) {
        minStack.push(value);
    }
}

private static void pop() {
    int removed = mainStack.pop();
    if (removed == minStack.peek()) {
        minStack.pop();
    }
}

private static void getMin() {
    System.out.println(minStack.peek());
}
```

---

### 면접 질문 1. 왜 보조 스택을 사용했나요?

#### 답변
매번 스택 전체를 순회해서 최솟값을 찾으면 O(n)이 됩니다.  
최솟값 전용 보조 스택을 유지하면 항상 top에 현재 최솟값이 있으므로 `peek()` 한 번으로 O(1)에 반환할 수 있습니다.

---

### 면접 질문 2. pop()할 때 minStack도 같이 pop하는 조건이 왜 `removed == minStack.peek()`인가요?

#### 답변
mainStack에서 제거된 값이 현재 최솟값과 같다면, 그 최솟값이 스택에서 사라진 것이므로 minStack에서도 제거해야 합니다.  
다음으로 작은 값이 minStack의 새로운 top이 되어 올바른 최솟값을 유지할 수 있습니다.

---

### 면접 질문 3. push할 때 `<=`를 쓴 이유는 무엇인가요?

#### 답변
같은 값이 여러 번 push될 수 있기 때문입니다.  
`<`만 쓰면 같은 최솟값이 두 번 push됐을 때 첫 번째 것만 minStack에 들어가고, pop()으로 하나를 제거하면 minStack도 함께 제거되어 남은 값이 최솟값인데도 minStack에 없는 상황이 발생합니다.  
`<=`를 사용하면 같은 값도 minStack에 중복 저장해서 이 문제를 방지할 수 있습니다.

---

### 회고
처음에 `getMin()`에서 스택 요소를 제거하면서 순회하는 O(n) 방식으로 접근했다.  
핵심은 조회와 제거를 구분하는 것이고, `peek()`이 조회 전용이라는 점을 확실히 익혔다.  
보조 스택 패턴은 면접에서 자주 나오는 패턴이므로 기억해두어야 한다.

## 문제 3 - 카드 뽑기 시뮬레이션

### 문제
1번부터 N번까지 카드가 순서대로 큐에 들어있을 때, 아래 규칙으로 카드를 뽑아 마지막에 남는 카드 번호를 출력하시오.

1. 맨 앞 카드를 버린다
2. 그 다음 맨 앞 카드를 맨 뒤로 보낸다
3. 카드가 1장 남을 때까지 반복한다

### 풀이 아이디어
1번부터 N번까지 카드를 순서대로 큐에 넣었다.  
큐의 크기가 1보다 클 때까지 반복하면서 `poll()`로 맨 앞 카드를 버리고, 다음 카드를 `poll()`로 꺼내 `add()`로 맨 뒤에 다시 넣었다.

### 사용한 개념
- `BufferedReader`
- `Queue<Integer>` / `LinkedList`
- `poll()` — 맨 앞에서 꺼내고 제거
- `add()` — 맨 뒤에 추가

### 시간복잡도
- O(n)

### 공간복잡도
- O(n)

---

### 버그 기록

#### 버그 1 — 변수명 `temp` 사용
```java
// 버그 코드
int temp = queue.getFirst();
queue.removeFirst();
queue.addLast(temp);

// 수정 코드 — removeFirst()가 꺼낸 값을 반환하므로 한 줄로 처리
queue.add(queue.poll());
```

#### 버그 2 — `Deque` 타입으로 선언
```java
// 버그 코드
Deque<Integer> queue = new LinkedList<>();

// 수정 코드 — 문제 조건에 맞게 Queue 타입 사용
Queue<Integer> queue = new LinkedList<>();
```

---

### 최종 코드
```java
Queue<Integer> queue = new LinkedList<>();

for (int i = 1; i <= n; i++) {
    queue.add(i);
}

while (queue.size() > 1) {
    queue.poll();                  // 맨 앞 버리기
    queue.add(queue.poll());       // 다음 카드 맨 뒤로
}

System.out.println(queue.peek());
```

---

### 심화 — 요세푸스 문제

이 문제는 요세푸스 문제의 변형이다.  
수학 공식을 사용하면 O(log n)으로 풀 수 있지만 구현 난이도가 높다.  
n이 최대 1000이므로 큐 시뮬레이션 O(n)으로 충분히 통과한다.

---

### 면접 질문 1. 이 코드가 전체적으로 어떻게 동작하는지 설명해보세요.

#### 답변
1번부터 N번까지 카드를 순서대로 큐에 넣었습니다.  
큐의 크기가 1이 될 때까지 반복하면서 `poll()`로 맨 앞 카드를 버리고, 다음 카드를 꺼내 `add()`로 맨 뒤에 다시 넣었습니다.  
카드가 1장 남으면 `peek()`으로 그 값을 출력했습니다.

---

### 면접 질문 2. `poll()`과 `peek()`의 차이는 무엇인가요?

#### 답변
`poll()`은 맨 앞 요소를 꺼내고 큐에서 제거합니다.  
`peek()`은 맨 앞 요소를 확인만 하고 큐에서 제거하지 않습니다.  
마지막에 남은 카드를 출력할 때는 제거할 필요가 없으므로 `peek()`을 사용했습니다.

---

### 면접 질문 3. 이 코드의 시간복잡도는 어떻게 되나요?

#### 답변
카드가 n장이고 매 루프마다 카드가 1장씩 제거되므로 루프는 n-1번 실행됩니다.  
각 루프에서 `poll()`, `poll()`, `add()`는 모두 O(1)이므로 전체 시간복잡도는 O(n)입니다.

---

### 회고
`Deque` 타입 대신 `Queue` 타입으로 선언해야 한다는 점과, 변수명은 역할이 드러나야 한다는 점을 다시 확인했다.  
`poll()` → 버리기, `add(queue.poll())` → 맨 뒤로 보내기 패턴이 큐 시뮬레이션의 핵심이다.  
카드 문제, 프린터 문제 모두 이 패턴으로 해결할 수 있다.
