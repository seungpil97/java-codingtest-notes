## 문제 1. 피보나치 수 (🟢)

### 문제 설명

n번째 피보나치 수를 재귀로 구하라. (백준 10870번)

### 풀이 아이디어

- 종료 조건: n==0 → 0, n==1 → 1
- 재귀: fibonacci(n) = fibonacci(n-1) + fibonacci(n-2)

### 사용한 개념

- 재귀 (분기형 — 호출이 2개로 퍼짐)

### 시간복잡도 / 공간복잡도

- 시간: O(2^n) — 중복 호출 발생
- 공간: O(n) — 스택 깊이

### 버그 기록

1. `return;` → int 메서드에서 컴파일 에러 (`return 0`으로 수정)
2. `fibonacci(n)` → n을 줄이지 않아 무한 루프 (`n-1`로 수정)
3. n==1 종료 조건 누락 → 음수 방향 무한 재귀

### 최종 코드

```java
private static int fibonacci(int n) {
  if (n == 0) return 0;
  if (n == 1) return 1;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```

### 💡 재귀 (Recursion)

**장점:**

- 수학적 점화식을 코드로 그대로 표현할 수 있어 가독성이 높다.
  피보나치처럼 f(n) = f(n-1) + f(n-2) 형태의 정의가 있는 문제에서 특히 직관적이다.
- 트리, DFS처럼 "같은 구조가 반복되는" 문제에서 반복문보다 짧고 명확한 코드가 나온다.

**단점:**

- 함수 호출마다 스택 프레임이 누적된다. n이 커지면 스택 오버플로우 위험이 있다.
  Java의 기본 스택 깊이는 약 수천 단계.
- 피보나치처럼 중복 호출이 발생하면 반복문보다 훨씬 느리다.
  이때는 메모이제이션(DP)으로 보완한다. (8주차에서 다룸)

**실무 사용처:**

- Linux `find` 명령어, Java `Files.walkFileTree()` — 폴더 안에 폴더가 있는 구조를 탐색할 때
  재귀가 자연스럽다. 몇 단계 깊이인지 몰라도 동일한 코드로 처리 가능하기 때문.
- Git 내부 커밋 히스토리 탐색 — 커밋은 부모 커밋을 가리키는 트리 구조이며,
  `git log`가 내부적으로 재귀 탐색을 사용한다.

### 질문 & 답변

- Q. n=0, n=1은 왜 고정값인가? 앞의 두 수가 없어서인가?
- A. 정확하다. 피보나치는 "앞의 두 항의 합"으로 정의되는데
n=0과 n=1은 앞에 두 항이 없어서 계산이 불가능하다.
그래서 시작값으로 정의로 주어진다.
재귀에서 종료 조건이 되는 이유도 같다 — 더 이상 쪼갤 수 없는 시점.

- Q. n=3의 앞의 두 수가 1+2라 결과가 3 아닌가?
- A. "앞의 두 수"는 인덱스가 아니라 피보나치 값이다.
fibonacci(2) = 1, fibonacci(1) = 1
→ fibonacci(3) = 1 + 1 = 2
fibonacci(2)의 값이 2가 아니라 1이기 때문에 3이 아니다.

| n | 계산                          | 값 |
   |---|-----------------------------|---|
| 0 | 정의                          | 0 |
| 1 | 정의                          | 1 |
| 2 | fibonacci(1) + fibonacci(0) | 1 |
| 3 | fibonacci(2) + fibonacci(1) | 2 |
| 4 | fibonacci(3) + fibonacci(2) | 3 |
| 5 | fibonacci(4) + fibonacci(3) | 5 |

### 회고

- 종료 조건을 2개(n==0, n==1) 모두 챙겨야 한다는 걸 버그를 통해 체득했다.
- 재귀 메서드에서 반환값이 있으면 반드시 return 값을 명시해야 한다.

## 문제 2. 재귀로 거듭제곱 구하기 (🟡)

### 문제 설명

base^exp를 재귀로 계산하라. (Math.pow 사용 금지)

### 풀이 아이디어

- 종료 조건: exp==0 → 1, exp==1 → base
- 재귀: getPower(base, exp) = base × getPower(base, exp-1)

### 사용한 개념

- 재귀 (선형형 — 호출이 1개로 이어짐)

### 시간복잡도 / 공간복잡도

- 시간: O(n)
- 공간: O(n) — 스택 깊이

### 버그 기록

없음 (1회 통과)

### 최종 코드

```java
private static int getPower(int base, int exp) {
  if (exp == 0) return 1;
  if (exp == 1) return base;
  return base * getPower(base, exp - 1);
}
```

### 💡 피보나치 vs 거듭제곱 — 재귀 구조 비교

```
// 피보나치 — 호출이 2개로 퍼짐 (트리형) → 중복 호출 발생
return fibonacci(n-1) + fibonacci(n-2);

// 거듭제곱 — 호출이 1개로 이어짐 (선형형) → 중복 없음
return base * getPower(base, exp-1);
```

호출이 2개로 퍼지면 중복 계산이 생기고, 1개로 이어지면 깔끔하다.
이 차이가 DP를 배울 때 핵심 포인트가 된다.

### 질문 & 답변

- Q. exp==1 종료 조건이 진짜 최적화인가?
- A. 사실상 최적화라고 부르기 어렵다. 호출 1번을 줄이는 수준이라 체감 불가.
진짜 최적화는 분할 정복으로 O(n) → O(log n)으로 줄이는 것이다.
예: 3^8 = 3^4 × 3^4 → 3^4만 계산하면 됨.
exp ≤ 20인 이 문제에서는 차이가 없고, 지금 단계 코드는 완벽히 맞다.

### 회고

- 재귀가 선형으로 이어지면 중복이 없어서 깔끔하다.
- exp==1 조건은 있어도 없어도 동작하지만, 불필요한 호출을 1번 줄여준다.

## 문제 3. 하노이 탑 (🟡)

### 문제 설명

n개의 원판을 1번→3번 기둥으로 이동. 총 횟수와 이동 순서 출력. (백준 11729번)

### 풀이 아이디어

- 1단계: (n-1)개를 from → via (보조 기둥 경유)
- 2단계: 가장 큰 원판 from → to (출력)
- 3단계: (n-1)개를 via → to
- 총 이동 횟수: 2^n - 1 → 비트 연산 `(1 << n) - 1`

### 사용한 개념

- 재귀 (분할 정복형)
- StringBuilder (출력 최적화)
- 비트 시프트 연산

### 시간복잡도 / 공간복잡도

- 시간: O(2^n)
- 공간: O(n) — 스택 깊이

### 버그 기록

1. 재귀 호출에서 n → n-1로 안 줄임 (무한 재귀)
2. n==1 블록과 2단계 출력문 누락

### 최종 코드

```java
public static void main(String[] args) throws IOException {
  BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
  int n = Integer.parseInt(br.readLine());

  StringBuilder sb = new StringBuilder();
  int count = (1 << n) - 1;
  sb.append(count).append("\n");

  hanoi(n, 1, 3, 2, sb);
  System.out.print(sb);
}

private static void hanoi(int n, int from, int to, int via, StringBuilder sb) {
  if (n == 1) {
    sb.append(from).append(" ").append(to).append("\n");
    return;
  }
  hanoi(n - 1, from, via, to, sb);
  sb.append(from).append(" ").append(to).append("\n");
  hanoi(n - 1, via, to, from, sb);
}
```

### 💡 하노이 탑 — 분할 정복 재귀

**장점:**

- "쪼개기 → 처리 → 합치기" 구조로 복잡한 문제를 3줄로 표현할 수 있다.
- 반복문으로 구현하면 스택을 직접 관리해야 해서 훨씬 복잡해진다.
  재귀는 콜 스택이 이 역할을 대신해준다.

**단점:**

- n=20이면 호출 횟수가 약 200만 번. 스택 깊이도 n에 비례해서 쌓인다.
  n이 커질수록 스택 오버플로우 위험이 있다.

**실무 사용처:**

- 컴파일러의 수식 파싱 트리 — 괄호 안에 괄호가 중첩된 수식을 분석할 때
  "쪼개기 → 처리 → 합치기" 재귀 패턴을 사용한다.
- 병합 정렬, 퀵 정렬 — 동일한 분할 정복 구조. 3주차에서 직접 구현한 병합 정렬이 같은 패턴.

### 질문 & 답변

- Q. 출력의 숫자는 원판 번호인가, 기둥 번호인가?
- A. 기둥 번호다. `from to` 형식으로, from 기둥에서 to 기둥으로 이동한다는 뜻.
예: `1 3` → 1번 기둥 맨 위 원판을 3번 기둥으로 이동.

### 회고

- 하노이의 핵심은 3단계 사고방식. 이걸 잡으면 코드는 짧다.
- StringBuilder는 n=20 기준 출력이 100만 번을 넘기 때문에 필수다.
- 비트 시프트 `(1 << n) - 1`로 2^n-1을 깔끔하게 표현할 수 있다.

- 피보나치(분기형) vs 거듭제곱(선형형) vs 하노이(분할 정복형) — 재귀의 3가지 형태를 모두 경험했다.

- 출력이 많을 때 StringBuilder를 쓰는 습관을 하노이에서 바로 적용했다.
