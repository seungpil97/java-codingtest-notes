## 문제 1. lower_bound 구현 🟢

### 문제 설명

정렬된 정수 배열과 목표값 `target`이 주어질 때, `target` 이상인 값이 처음 등장하는 인덱스를 반환한다.
`target`보다 큰 값이 없으면 배열의 길이를 반환한다.

### 풀이 아이디어

- `arr[mid] >= target`이면 `mid`가 정답 후보 → `right = mid`로 유지
- `arr[mid] < target`이면 정답은 오른쪽 → `left = mid + 1`
- 탐색이 끝나면 `left == right`가 첫 번째 정답 인덱스

### 사용한 개념

- 이분 탐색 (Binary Search)

### 시간복잡도 / 공간복잡도

|    | 복잡도      |
|----|----------|
| 시간 | O(log N) |
| 공간 | O(1)     |

### 버그 기록

1. `arr[mid] > target` 조건에서 `arr[mid - 1]`을 반환 → **인덱스가 아닌 값을 반환**하는 버그
2. `arr[mid] == target`일 때 `mid` 자체를 정답 후보로 기억하지 않고 버림
3. `right = n - 1`로 시작 → `target`이 없을 때 `n`(배열 길이)을 반환할 수 없음

### 최종 코드

```java
public class April01LowerBound {
  static int[] arr;

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    int n = Integer.parseInt(br.readLine());
    StringTokenizer st = new StringTokenizer(br.readLine());

    arr = new int[n];
    for (int i = 0; i < n; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }

    int target = Integer.parseInt(br.readLine());
    System.out.println(findLowerBound(0, n, target));
  }

  private static int findLowerBound(int left, int right, int target) {
    while (left < right) {
      int mid = (left + right) / 2;
      if (arr[mid] < target) {
        left = mid + 1;
      } else {
        right = mid;
      }
    }
    return left;
  }
}
```

### 💡 이분 탐색 (Binary Search)

**장점:**

- 정렬된 배열에서 O(log N)으로 탐색 가능. N = 1,000,000이어도 최대 20번만 비교하면 됨. 선형 탐색 O(N) 대비
  압도적으로 빠름

**단점:**

- 정렬이 전제 조건이라 삽입/삭제가 빈번한 데이터에는 적합하지 않음. 매번 정렬 비용이 들어가면 O(N log N)으로 오히려 손해

**실무 사용처:**

- **MySQL** — B+Tree 인덱스 내부가 이분 탐색 기반. `WHERE id = 100` 같은 쿼리가 O(log N)으로 동작하는
  이유
- **Git** — `git bisect` 명령어가 커밋 이력에서 버그 발생 시점을 이분 탐색으로 찾음
- **Java** — `Arrays.binarySearch()`, `Collections.binarySearch()` 내부 구현

### 질문 & 답변

**Q. `right = n`으로 시작해야 하는 이유는?**

`target`이 배열의 모든 원소보다 크면 정답은 `n`(배열 길이)이다.
`right = n - 1`로 시작하면 `left`가 `n`까지 올라갈 수 없어서 이 케이스를 처리할 수 없다.

**Q. `right = mid`로 두는 이유는?**

`arr[mid] >= target`일 때 `mid`가 정답일 수 있으므로 버리면 안 된다.
`right = mid - 1`로 두면 정답을 놓칠 수 있다.

### 회고

- 처음에 인덱스 대신 값을 반환하는 실수를 했다. `lower_bound`는 항상 **인덱스**를 반환한다는 것을 기억하자
- `right`의 초기값을 `n`으로 설정하는 이유를 이해하는 것이 핵심이었다

## 문제 2. 값의 개수 세기 🟡

### 문제 설명

정렬된 정수 배열에서 특정 값 `target`이 몇 번 등장하는지 O(log N)으로 구하라.

### 풀이 아이디어

- `lower_bound(target)` : `target` 이상이 처음 등장하는 인덱스
- `upper_bound(target)` : `target` 초과가 처음 등장하는 인덱스
- 개수 = `upper_bound - lower_bound`

```
배열: [1, 2, 2, 2, 3, 4, 5]
         ↑           ↑
findLowerBound(2) = 1
findUpperBound(2) = 4

4 - 1 = 3 ✅
```

### 사용한 개념

- lower_bound / upper_bound 패턴

### 시간복잡도 / 공간복잡도

|    | 복잡도      |
|----|----------|
| 시간 | O(log N) |
| 공간 | O(1)     |

### 버그 기록

1. 이분 탐색 내부에서 `arr[mid] == target`이면 `count++` 시도 → 이분 탐색은 모든 원소를 방문하지 않으므로 일부
   target을 놓침

### 최종 코드

```java
public class April01CountTarget {
  static int[] arr;

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    int n = Integer.parseInt(br.readLine());
    StringTokenizer st = new StringTokenizer(br.readLine());

    arr = new int[n];
    for (int i = 0; i < n; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }

    int target = Integer.parseInt(br.readLine());

    int lowerIndex = findLowerBound(0, n, target);
    int upperIndex = findUpperBound(0, n, target);

    System.out.println(upperIndex - lowerIndex);
  }

  private static int findLowerBound(int left, int right, int target) {
    while (left < right) {
      int mid = (left + right) / 2;
      if (arr[mid] < target) left = mid + 1;
      else right = mid;
    }
    return left;
  }

  private static int findUpperBound(int left, int right, int target) {
    while (left < right) {
      int mid = (left + right) / 2;
      if (arr[mid] <= target) left = mid + 1;
      else right = mid;
    }
    return left;
  }
}
```

### 💡 lower_bound vs upper_bound 비교

|                  | 조건                   | 의미             |
|------------------|----------------------|----------------|
| `findLowerBound` | `arr[mid] < target`  | target 미만이면 버림 |
| `findUpperBound` | `arr[mid] <= target` | target 이하이면 버림 |

조건 하나(`<` vs `<=`) 차이로 동작이 완전히 달라진다.

### 질문 & 답변

**Q. 왜 이분 탐색 안에서 count++로 세면 안 되나?**

이분 탐색은 매 단계마다 탐색 범위를 절반씩 버리기 때문에 `target`과 같은 모든 원소를 방문하지 않는다.
예를 들어 `[2, 2, 2]`에서 중간 원소 하나만 방문하고 종료될 수 있다.

### 회고

- 처음 접근이 완전히 틀렸다. 이분 탐색은 "탐색"이지 "순회"가 아니라는 걸 다시 확인했다
- `lower_bound`와 `upper_bound`의 조건 차이(`<` vs `<=`) 하나를 외워두면 개수 세기 문제는 빠르게 풀 수 있다

## 문제 3. 나무 자르기 🟡

### 문제 설명

절단기 높이 `H`를 설정하면 `H`보다 높은 나무는 `(나무 높이 - H)`만큼 잘려 나온다.
상근이가 최소 `M`미터를 가져갈 수 있는 절단기 높이의 최댓값을 구하라. (백준 2805번)

### 풀이 아이디어

- `H`를 높일수록 얻는 나무가 줄어드는 **단조성** 존재
- **답(H)의 범위**에 이분 탐색 적용 (parametric search)
- `calculateWood(mid) >= M`이면 정답 후보 저장 후 `left = mid + 1` (더 높여볼 수 있다)
- `calculateWood(mid) < M`이면 너무 높다 → `right = mid - 1`

```
탐색 범위: left = 0, right = 나무 최댓값
정답 후보: wood(mid) >= M 일 때마다 answer 갱신
```

### 사용한 개념

- parametric search (매개변수 탐색)

### 시간복잡도 / 공간복잡도

|    | 복잡도                    |
|----|------------------------|
| 시간 | O(N log H) — H: 나무 최댓값 |
| 공간 | O(N)                   |

### 버그 기록

1. 이분 탐색 없이 `h++` / `h / 2`로 탐색 → `sum == m`이 정확히 맞는 순간이 오지 않으면 무한 루프
2. `sum += arr[i] - h` — `arr[i] < h`이면 음수가 합산됨 → `Math.max(0, arr[i] - h)` 필요
3. `int m` 선언 — M은 최대 2,000,000,000으로 int 범위를 초과할 수 있음 → `long` 사용

### 최종 코드

```java
public class April01TreeCut {
  static int[] arr;

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
    StringTokenizer st = new StringTokenizer(br.readLine());
    int n = Integer.parseInt(st.nextToken());
    long m = Long.parseLong(st.nextToken());

    arr = new int[n];
    st = new StringTokenizer(br.readLine());
    for (int i = 0; i < n; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }

    System.out.println(findMaxHeight(m));
  }

  private static long findMaxHeight(long m) {
    long left = 0;
    long right = 0;
    for (int treeHeight : arr) {
      right = Math.max(right, treeHeight);
    }
    long answer = 0;

    while (left <= right) {
      long mid = (left + right) / 2;
      long wood = calculateWood(mid);

      if (wood >= m) {
        answer = mid;
        left = mid + 1;
      } else {
        right = mid - 1;
      }
    }

    return answer;
  }

  private static long calculateWood(long height) {
    long total = 0;
    for (int treeHeight : arr) {
      if (treeHeight > height) {
        total += treeHeight - height;
      }
    }
    return total;
  }
}
```

### 💡 parametric search (매개변수 탐색)

**장점:**

- "답이 될 수 있는 범위"에 이분 탐색을 적용해서 O(N log H)로 해결 가능. 완전 탐색 O(N * H)와 비교하면 H =
  1,000,000,000일 때 30배 이상 차이남

**단점:**

- 단조성(조건을 만족하는 구간이 연속적이어야 함)이 없으면 적용 불가. 문제를 보고 단조성 여부를 먼저 판단해야 함

**실무 사용처:**

- **배포 시스템** — 특정 트래픽 임계값에서 응답시간이 급격히 늘어날 때 최적 배포 단위를 이분 탐색으로 결정
- **DB 쿼리 최적화** — `LIMIT` 값을 이분 탐색으로 조정해 최적 페이지 크기 탐색

### 질문 & 답변

**Q. 일반 이분 탐색과 parametric search의 차이는?**

|       | 일반 이분 탐색             | parametric search |
|-------|----------------------|-------------------|
| 탐색 대상 | 배열의 값                | 답의 범위             |
| 조건    | `arr[mid] == target` | `f(mid) >= 조건`    |
| 예시    | lower_bound, 수 찾기    | 나무 자르기, 랜선 자르기    |

**Q. `Arrays.stream(arr).max().getAsInt()`가 무엇인가?**

배열을 `IntStream`으로 변환 후 최댓값을 찾는 코드다.
`.max()`가 `OptionalInt`를 반환하는 이유는 배열이 비어있을 수 있기 때문이다.
순회 한 번으로 최댓값을 구하는 `for-each` 방식이 O(N)으로 동일하고 더 직관적이다.

### 회고

- 처음에 이분 탐색을 쓰지 않고 `h++` / `h/2`로 접근했는데 이는 parametric search의 핵심인 "답의 범위에 이분 탐색"
  을 적용하지 않은 것이다
- `long` 범위 주의가 이번 문제의 핵심 함정이었다. M과 합산값 모두 `long`으로 처리해야 한다
- `Math.max(0, arr[i] - h)` 처리를 빠뜨리지 않도록 주의하자

## 오늘의 핵심 정리

```
lower_bound   : arr[mid] < target  → left = mid + 1  (< 조건)
upper_bound   : arr[mid] <= target → left = mid + 1  (<= 조건)
개수 세기     : upper_bound - lower_bound
parametric    : 답의 범위에 이분 탐색 + 단조성 판단이 핵심
```
