## 문제 1. 재귀로 배열 최솟값 찾기

### 문제 설명

정수 배열 `{5, 3, 8, 1, 9, 2, 7}`에서 재귀를 사용해 최솟값을 구한다.
`for문`, `Arrays.stream` 사용 금지.

### 풀이 아이디어

`arr[0..n-1]`의 최솟값 = `Math.min(arr[n-1], arr[0..n-2]의 최솟값)` 정의를 그대로 코드로 옮긴다.

- 기저 조건: `n == 1` → `arr[0]` 반환
- 재귀 단계: `Math.min(arr[n-1], findMin(n-1))`

### 사용한 개념

- 재귀 (반환값 설계)
- 수학적 귀납법적 사고 (n개 문제를 n-1개 문제로 축소)

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(N) — 배열 원소를 한 번씩 방문
- 공간복잡도: O(N) — 재귀 호출 스택 깊이 N

### 버그 기록

| 버그                        | 원인                              | 수정                                |
|---------------------------|---------------------------------|-----------------------------------|
| 입력 순서 역전                  | `StringTokenizer`를 `n` 파싱 전에 생성 | 문제 조건상 하드코딩이므로 입력 코드 전체 제거        |
| 파라미터 3개 (min, current, n) | 반환값으로 처리 가능한 상태를 파라미터로 전달       | `findMin(int n)` 단일 파라미터 + int 반환 |
| 기저 조건 `n == 0`            | `arr[-1]` 접근 위험                 | `n == 1`로 변경                      |

### 최종 코드

```java
package week4;

public class March31RecursiveMin {

  static int[] arr = {5, 3, 8, 1, 9, 2, 7};

  public static void main(String[] args) {
    System.out.println("최솟값: " + findMin(arr.length));
  }

  private static int findMin(int n) {
    if (n == 1) return arr[0];
    return Math.min(arr[n - 1], findMin(n - 1));
  }
}
```

### 💡 재귀 반환값 설계 패턴

**장점:**

- 상태를 파라미터로 들고 다니지 않아도 되므로 함수 시그니처가 단순해진다.
  파라미터가 많아질수록 호출부에서 실수하기 쉽고, 테스트도 어려워진다.
- "이번 단계의 결과"와 "나머지를 재귀에게 위임"하는 두 가지 역할이 명확하게 분리된다.

**단점:**

- 반환값이 없는 `void` 재귀(출력, 상태 변경이 목적)에는 적용하기 어렵다.
  이 경우 `StringBuilder`를 파라미터로 전달하거나 전역 변수를 사용하는 패턴이 대안이다.

**실무 사용처:**

- **Java Stream reduce()**: `stream.reduce(Integer::min)` 내부가 이 패턴과 동일하다.
  `reduce`는 "현재 원소와 나머지 축소 결과를 합치는" 재귀 구조를 반복문으로 풀어낸 것이다.

### Q&A

**Q. 왜 기저 조건이 `n == 0`이 아니라 `n == 1`인가?**

`n == 0`이면 `arr[0-1] = arr[-1]`에 접근하게 되어 `ArrayIndexOutOfBoundsException`이 발생한다.
"원소가 1개일 때 그 자체가 최솟값"이라는 정의가 더 자연스럽고 안전하다.

**Q. 재귀 설계 시 어떤 순서로 생각하면 좋은가?**

1. 기저 조건 (언제 멈추는가?)
2. 이번 단계에서 할 일 (현재 원소 처리)
3. 나머지를 재귀에 위임 (n-1로 축소)

---

### 회고

- 파라미터로 상태를 전달하기 전에, 반환값으로 해결할 수 있는지 먼저 확인하는 습관이 필요하다.
- 문제의 힌트 정의(`arr[0..n-1]의 최솟값 = Math.min(arr[n-1], arr[0..n-2]의 최솟값)`)를 그대로 코드로
  옮기는 연습이 재귀의 핵심임을 느꼈다.

## 문제 2. 이분 탐색으로 위치 찾기

### 문제 설명

오름차순 정렬된 배열에서 `target`을 이분 탐색으로 찾아 인덱스를 출력한다.
없으면 `-1` 출력. `Arrays.binarySearch()` 사용 금지.

### 풀이 아이디어

- `left`, `right`로 탐색 범위를 관리한다.
- `mid = (left + right) / 2` 계산 후 세 가지 경우로 분기한다.
    - `arr[mid] == target` → 반환
    - `arr[mid] < target` → 오른쪽 절반 탐색 (`left = mid + 1`)
    - `arr[mid] > target` → 왼쪽 절반 탐색 (`right = mid - 1`)
- `left > right`가 되면 값이 없으므로 `-1` 반환

### 사용한 개념

- 이분 탐색 (Binary Search)
- 재귀 반환값 설계

### 시간복잡도 / 공간복잡도

- 시간복잡도: O(log N) — 매 단계마다 탐색 범위 절반 감소
- 공간복잡도: O(log N) — 재귀 호출 스택 깊이

### 버그 기록

| 버그                        | 원인                                              | 수정                             |
|---------------------------|-------------------------------------------------|--------------------------------|
| 선형 탐색 O(N)                | `findNumber(n-1, find)` 로 인덱스를 1씩 줄임, mid 계산 없음 | `left`, `right`, `mid` 구조로 재설계 |
| `Arrays.sort()` 불필요       | 문제 조건: "오름차순 정렬 보장"                             | 제거                             |
| `import java.util.Arrays` | sort 제거 후 미사용 import                            | 제거                             |
| 변수명 `find`                | 동사형, 역할 불명확                                     | `target`으로 변경                  |

### 최종 코드

```java
package week4;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.StringTokenizer;

public class March31BinarySearch {

  static int[] arr;

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

    int n = Integer.parseInt(br.readLine());
    StringTokenizer st = new StringTokenizer(br.readLine());
    arr = new int[n];
    for (int i = 0; i < n; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }
    int target = Integer.parseInt(br.readLine());

    System.out.println(binarySearch(0, n - 1, target));
  }

  private static int binarySearch(int left, int right, int target) {
    if (left > right) return -1;

    int mid = (left + right) / 2;

    if (arr[mid] == target) return mid;
    if (arr[mid] < target) return binarySearch(mid + 1, right, target);
    return binarySearch(left, mid - 1, target);
  }
}
```

### 💡 이분 탐색 (Binary Search)

**장점:**

- 탐색 범위를 매 단계 절반으로 줄이므로 O(log N)으로 동작한다.
  N = 100,000이어도 최대 17번 비교로 탐색이 끝난다.
  선형 탐색 O(N) 대비 수백 배 빠르다.

**단점:**

- 반드시 정렬된 배열에서만 사용 가능하다.
  정렬 비용 O(N log N)이 선행되어야 하므로, 탐색을 딱 1번만 할 거라면 선형 탐색이 오히려 더 빠를 수 있다.
  탐색 횟수가 많을수록 이분 탐색이 유리하다.

**실무 사용처:**

- **MySQL B-Tree 인덱스**: 정렬된 키 구조에서 이분 탐색으로 레코드를 찾는다. `WHERE id = 1000` 쿼리가 풀스캔
  없이 빠른 이유다.
- **Git bisect**: 수천 개의 커밋 중 버그가 생긴 커밋을 이분 탐색으로 좁혀나간다. 커밋 1,000개도 10번이면 특정된다.
- **Java `Arrays.binarySearch()`**: 오늘 구현한 알고리즘과 동일한 원리로 동작하는 표준 라이브러리다.

### Q&A

**Q. `mid = (left + right) / 2`에서 오버플로우가 발생할 수 있는가?**

`left`와 `right`가 모두 `int` 최댓값에 가까울 경우 `left + right`가 오버플로우될 수 있다.
안전한 작성법은 `mid = left + (right - left) / 2`이다.
코딩테스트 범위에서는 대부분 문제없지만, 실무나 대용량 데이터를 다룰 때는 이 방식을 쓰는 것이 좋다.

**Q. 이분 탐색을 반복문(while)으로도 구현할 수 있는가?**

```java
while(left <=right){
int mid = left + (right - left) / 2;
    if(arr[mid]==target)return mid;
    if(arr[mid] <target)left =mid +1;
  else right =mid -1;
  }
  return-1;
```

반복문 버전은 재귀 스택이 없으므로 공간복잡도가 O(1)이다.
실무에서는 반복문 버전을 더 많이 사용한다.

### 회고

- 인덱스를 1씩 줄이는 것과 절반씩 줄이는 것의 차이가 O(N)과 O(log N)의 차이임을 직접 확인했다.
- `left > right` 기저 조건의 의미: 탐색 범위가 소진됐다는 뜻이고, 이 시점에서 값이 없음이 확정된다.

## 문제 3. 수 찾기 (백준 1920번)

### 문제 설명

N개의 정수 배열 A가 주어진다.
M개의 질의값이 각각 배열 A에 존재하면 `1`, 없으면 `0`을 출력한다.
`HashSet` 금지 — 이분 탐색으로 구현.

### 풀이 아이디어

1. 배열 A를 `Arrays.sort()`로 정렬한다.
2. 질의값 M개를 `for문`으로 순회하면서 각각에 대해 이분 탐색을 수행한다.
3. 이분 탐색 결과를 `StringBuilder`에 `append`한다.

### 사용한 개념

- 이분 탐색 (문제 2와 동일 구조, 반환값만 인덱스 → 0/1로 변경)
- `Arrays.sort()`
- `StringBuilder` (N ≥ 100,000이므로 `println` 반복 금지)

### 시간복잡도 / 공간복잡도

- 시간복잡도: O((N + M) log N) — 정렬 O(N log N) + M번 이분 탐색 O(M log N)
- 공간복잡도: O(N + M)

### 버그 기록

| 버그                          | 원인                              | 수정                                 |
|-----------------------------|---------------------------------|------------------------------------|
| 이분 탐색 없음                    | 두 배열을 동시에 재귀로 줄이는 선형 비교 구조      | 배열 정렬 후 질의마다 `binarySearch` 호출     |
| `return` 없는 기저 조건           | 조건 분기 후 실행 계속 진행 → 무한 재귀        | `left > right` 기저 조건 + 즉시 `return` |
| `sb.reverse()`              | 재귀 방향 오류를 역전으로 보정 시도            | `for문` 순방향 순회로 해결, `reverse()` 제거  |
| `Arrays.sort()` 누락          | 정렬 없이 이분 탐색 시도 (이분 탐색 전제 조건 위반) | `Arrays.sort(arr)` 추가              |
| 변수명 `firstArr`, `secondArr` | 역할 불명확                          | `arr`, `queries` 또는 `for문` 내 지역변수  |

### 최종 코드

```java
package week4;

import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;
import java.util.Arrays;
import java.util.StringTokenizer;

public class March31NumberSearch {

  static int[] arr;

  public static void main(String[] args) throws IOException {
    BufferedReader br = new BufferedReader(new InputStreamReader(System.in));

    int arrSize = Integer.parseInt(br.readLine());
    StringTokenizer st = new StringTokenizer(br.readLine());
    arr = new int[arrSize];
    for (int i = 0; i < arrSize; i++) {
      arr[i] = Integer.parseInt(st.nextToken());
    }
    Arrays.sort(arr);

    int queryCount = Integer.parseInt(br.readLine());
    st = new StringTokenizer(br.readLine());
    StringBuilder sb = new StringBuilder();

    for (int i = 0; i < queryCount; i++) {
      int target = Integer.parseInt(st.nextToken());
      sb.append(binarySearch(0, arrSize - 1, target)).append('\n');
    }

    System.out.print(sb);
  }

  private static int binarySearch(int left, int right, int target) {
    if (left > right) return 0;

    int mid = (left + right) / 2;

    if (arr[mid] == target) return 1;
    if (arr[mid] < target) return binarySearch(mid + 1, right, target);
    return binarySearch(left, mid - 1, target);
  }
}
```

### Q&A

**Q. `HashSet`을 쓰면 O(1) 탐색인데, 왜 이분 탐색을 배우는가?**

`HashSet`은 평균 O(1)이지만, 해시 충돌 시 O(N)으로 악화될 수 있다.
이분 탐색은 최악의 경우에도 O(log N)을 보장하며, 정렬된 데이터의 범위 탐색(`lower_bound`, `upper_bound`)에도
응용된다.
또한 `HashSet`이 허용되지 않는 코딩테스트 조건에서도 대응할 수 있다.

**Q. `StringBuilder`에 `'\n'`을 쓰는 이유는?**

`"\n"`(String)보다 `'\n'`(char)이 `append` 오버로드 측면에서 가볍다.
N이 100,000인 경우 `System.out.println()` 반복 호출은 I/O 비용이 크므로,
`StringBuilder`에 모아서 한 번에 출력하는 패턴이 필수다.

- 두 배열을 동시에 재귀로 순회하는 것이 이분 탐색이 아님을 명확히 이해했다.
- 이분 탐색의 구조는 문제 2와 동일하고, 반환값만 "인덱스"에서 "0/1"로 바꾸면 된다.
  자료구조/알고리즘의 재사용성을 체감한 문제였다.
- `return`이 없는 기저 조건이 얼마나 치명적인지 다시 확인했다.

## 오늘의 전체 회고

이분 탐색의 핵심은 **"절반을 버린다"** 는 결정이다.
단순히 인덱스를 1씩 줄이는 것과 절반씩 줄이는 것의 차이가 O(N)과 O(log N)을 만든다.
문제 2와 문제 3이 동일한 `binarySearch` 구조를 공유한다는 점에서,
알고리즘을 잘 설계하면 재사용이 가능하다는 것도 배웠다.
